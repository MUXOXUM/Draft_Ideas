# Правила для новых проектов

Этот репозиторий развивается как коллекция самостоятельных мини-проектов на чистых HTML, CSS и JS для статического сервера. Цель этого документа: сохранить единообразие кода, упростить сопровождение и сделать будущие проекты предсказуемыми по структуре.

## 1. Минимальная структура папки проекта

У каждого проекта должна быть отдельная директория в корне репозитория.

Минимум:

```text
project_name/
  index.html
  README.md
```

### `index.html`

Минимальный исполняемый вход проекта.

Требования:

- путь к `favicon.ico` должен быть относительным и вести на корневой favicon;
- интерфейс должен быть адаптивным и корректно работать как минимум на desktop и mobile-экранах;
- при наличии внешних данных должны быть предусмотрены fallback-сценарии.

### `README.md`

У каждого проекта README должен быть коротким, но содержательным. Базовый шаблон:

```md
# Название проекта

Краткая идея проекта в 1-3 предложениях.

Нужно объяснить:

- в чем идея проекта;
- что он демонстрирует;
- почему он может быть интересен.

## Как выглядит сайт

Короткое описание визуальной идеи, атмосферы или того, что пользователь видит на экране.

Нужно описать:

- что пользователь видит сразу после открытия;
- как устроена композиция экрана;
- есть ли режимы, экраны или состояния интерфейса.

## Управление

Нужно перечислить:

- клавиши;
- мышь;
- колесо;
- жесты;
- режимы взаимодействия.

## Интересные решения

Нужно отмечать только действительно содержательные вещи:

- нестандартный рендер;
- математическая модель;
- оптимизация;
- необычная навигация;
- интересная подача данных;
- ограничения, которые были обойдены красивым способом.
```

## 2. Общие правила

- Название папки проекта: `snake_case`, короткое и понятное.
- Название проекта в интерфейсе и в `projects.json`: человекочитаемое.
- Внутри одного проекта имена файлов должны быть предсказуемыми: `app.js`, `app.css`, `images/.`.

- не перегружать стартовую загрузку тяжелыми ассетами без необходимости;
- не загружать заранее все ресурсы, если можно подгружать их по мере необходимости;
- критически важный интерфейс должен появляться быстро даже без второстепенных данных;
- если проект зависит от сети, у пользователя должно быть понятное состояние загрузки или fallback-поведение.


## 3. Общие UI-концепции и соглашения по управлению

### Для 3D-сцен и камер

Если проект предполагает свободный просмотр 3D сцены, желательно использовать схему:

- `LMB` или основное перетаскивание: `arcball`-вращение;
- `RMB` или `MMB`: `pan`;
- колесо мыши: `zoom`;
- двойной клик: сброс вида, если это уместно;


### Для проектов с несколькими экранами, вариантами или слайдами

Базовое соглашение по навигации:

- `ArrowLeft` и `ArrowRight`;
- `A` и `D` и `Ф` и `Ы`;
