# Правила для новых проектов

Этот репозиторий развивается как коллекция самостоятельных мини-проектов на чистых HTML, CSS и JS. Цель этого документа: сохранить единообразие кода, упростить сопровождение и сделать будущие проекты предсказуемыми по структуре.

## 1. Общий принцип

Каждый новый проект должен быть:

- самостоятельным;
- статически обслуживаемым;
- понятным без внешней сборки;
- оформленным так, чтобы его можно было легко добавить в каталог на главной странице.

Проект должен нормально работать как часть обычного static server. Нужно исходить из того, что здесь нет backend-логики, серверных шаблонов, баз данных и сложной сборочной инфраструктуры.

## 2. Минимальная структура папки проекта

У каждого проекта должна быть отдельная директория в корне репозитория.

Минимум:

```text
project_name/
  index.html
  README.md
```

Рекомендуемая структура для более крупных проектов:

```text
project_name/
  index.html
  README.md
  styles/
    app.css
  scripts/
    app.js
  images/
    ...
  references/
    ...
```

## 3. Что обязательно должно быть

### `index.html`

Минимальный исполняемый вход проекта.

Требования:

- файл должен открывать сам проект, а не заглушку;
- путь к `favicon.ico` должен быть относительным и вести на корневой favicon;
- нужно учитывать мобильные экраны через `meta viewport`;
- интерфейс должен быть адаптивным и корректно работать как минимум на desktop и mobile-экранах;
- проект не должен зависеть от backend-роутов;
- при наличии внешних данных должны быть предусмотрены fallback-сценарии.

### `README.md`

Краткое и понятное описание проекта для каталога и будущей поддержки.

README обязателен даже для маленьких экспериментов.

## 4. Рекомендации по организации файлов

- Если проект становится больше примерно чем на один короткий `index.html`, CSS и JS лучше выносить в отдельные файлы.
- Встроенные `<style>` и `<script>` допустимы только для совсем маленьких и одноразовых экспериментов.
- Если в проекте есть изображения, текстуры или другие ассеты, они должны лежать в отдельной папке.
- Если есть референсы, наброски или исходники для вдохновения, их стоит складывать в `references/`, а не смешивать с финальными файлами проекта.

## 5. Именование

- Название папки проекта: `snake_case`, короткое и понятное.
- Название проекта в интерфейсе и в `projects.json`: человекочитаемое.
- Внутри одного проекта имена файлов должны быть предсказуемыми: `app.js`, `app.css`, `images/...`.

## 6. Обязательная структура README

У каждого проекта README должен быть коротким, но содержательным. Базовый шаблон:

```md
# Название проекта

Краткая идея проекта в 1-3 предложениях.

## Как выглядит сайт

Короткое описание визуальной идеи, атмосферы или того, что пользователь видит на экране.

## Управление

- ...
- ...

## Интересные решения

- ...
- ...
```

Если какой-то раздел не нужен, его можно опустить, но желательно придерживаться именно этой структуры.

### Что писать в README

#### Краткая идея

Нужно объяснить:

- в чем идея проекта;
- что он демонстрирует;
- почему он может быть интересен.

#### Как выглядит сайт

Нужно описать:

- что пользователь видит сразу после открытия;
- как устроена композиция экрана;
- есть ли режимы, экраны или состояния интерфейса.

#### Управление

Нужно перечислить:

- клавиши;
- мышь;
- колесо;
- жесты;
- режимы взаимодействия.

#### Интересные решения

Нужно отмечать только действительно содержательные вещи:

- нестандартный рендер;
- математическая модель;
- оптимизация;
- необычная навигация;
- интересная подача данных;
- ограничения, которые были обойдены красивым способом.

## 7. Общие UI-концепции и соглашения по управлению

Чтобы проекты ощущались частями одной коллекции, нужно по возможности переиспользовать одинаковые паттерны управления.

### Для 3D-сцен и камер

Если проект предполагает свободный просмотр сцены, желательно использовать знакомую схему:

- `LMB` или основное перетаскивание: `arcball`-вращение;
- `RMB` или `MMB`: `pan`;
- колесо мыши: `zoom`;
- двойной клик: сброс вида, если это уместно;
- при активном перетаскивании курсор должен явно менять состояние.

Если проект отходит от этой схемы, причина должна быть осознанной, а не случайной.

### Для проектов с несколькими экранами, вариантами или слайдами

Базовое соглашение по навигации:

- `ArrowLeft` и `ArrowRight`;
- `A` и `D`;
- при необходимости можно поддерживать русские раскладки, если это не усложняет код.

### Для интерфейсных подсказок

- если управление неочевидное, на экране должна быть короткая подсказка;
- подсказка должна быть компактной и не мешать основной сцене;
- если есть режим камеры или специальный режим управления, это нужно явно показывать.

## 8. Ограничения статического сервера

Все проекты в этом репозитории нужно проектировать с учетом ограничений static hosting.

Это значит:

- нельзя рассчитывать на backend;
- нельзя рассчитывать на серверную обработку файлов;
- нельзя рассчитывать на приватные API-ключи в клиентском коде;
- внешние API допустимы только если проект без них не ломается полностью;
- при сетевых ошибках должны быть fallback-состояния или понятные сообщения;
- все пути к локальным файлам должны быть относительными и совместимыми с публикацией из поддиректории.

Нужно помнить, что локальное открытие через `file://` и запуск через static server ведут себя по-разному. Если проект использует `fetch`, модальные README, JSON-файлы или внешние ресурсы, это следует учитывать и по возможности пояснять в интерфейсе или README.

## 9. Что желательно соблюдать в коде

- Код должен быть достаточно простым, чтобы через время его можно было быстро перечитать.
- Повторяющиеся паттерны внутри проекта лучше выносить в отдельные функции.
- Если один и тот же паттерн начинает повторяться между проектами, стоит думать о shared-утилитах.
- Не стоит добавлять сложную инфраструктуру без реальной необходимости.
- Если проект вычислительно тяжелый, нужно продумать поведение на слабых устройствах.

## 10. Адаптивный дизайн

Адаптивность для новых проектов обязательна.

Это значит:

- проект должен оставаться читаемым и управляемым на desktop и mobile;
- интерфейс не должен ломаться на узких экранах;
- элементы управления, подписи и подсказки не должны перекрывать критически важный контент;
- если проект визуально сложный, нужно продумывать отдельное поведение для маленьких экранов;
- размеры, отступы и типографика должны подстраиваться под ширину и высоту экрана;
- heavy-сцены и сложный рендер желательно адаптировать под слабые устройства и мобильные браузеры.

## 11. Перед добавлением нового проекта

Минимальный чеклист:

- есть отдельная папка проекта;
- есть рабочий `index.html`;
- есть `README.md`;
- проект работает как статическая страница;
- дизайн адаптивен;
- управление описано в README;
- если есть нестандартные решения, они коротко описаны;
- проект добавлен в `projects.json`.

## 12. Когда можно отступать от правил

Эти правила нужны не ради формальности, а ради системности.

Отступать можно, если:

- это делает проект заметно лучше;
- это уменьшает сложность, а не увеличивает ее;
- это осознанное исключение, а не случайность.

Если проект нарушает базовый шаблон, желательно чтобы причина была понятна из структуры или README.
